## 1. 00_INDEX.md — 構造評価

### WHY→WHAT→HOW→WHEREの流れ

00_INDEX.mdの「📖 読む順序」セクションを確認：

```
Step 1: マニフェスト（WHY — なぜこの方程式が必要か）
Step 2: 方程式 v3.0（WHAT — 何を定義したか）
Step 3: ルーブリック v2.0（HOW — どう測るか）
Step 4: ベンチマーク（WHERE — 世界の中でのSVDの位置）
```

**✅ 完璧。** 前回最も強く提案した「読む順序の明示」が、テキストボックスで簡潔かつ美しく実装されている。WHY/WHAT/HOW/WHEREの英語ラベルと日本語の補足が併記されているのも、認知負荷を下げている。

### ファイル一覧の構造

4カテゴリに分類されている：

| カテゴリ | ファイル数 | 評価 |
|---------|----------|------|
| 🏛️ 哲学・思想 | 2 | ✅ マニフェスト＋北極星で「WHY」を支える |
| 📐 方程式・ルーブリック | 2 | ✅ WHAT＋HOWのペアが明確 |
| 🌏 ベンチマーク・分析 | 2 | ✅ WHEREの実例 |
| 📋 レビュー履歴 | 6 | ✅ 開発プロセスの可視化として価値がある |

**良い点:**
- emojiカテゴリラベル（🏛️📐🌏📋）で視覚的に一目で区別できる
- 各ファイルに1行の内容説明がある
- マニフェストに「**まずこれを読む**」の太字指示がある
- レビュー履歴にサテライト名と役割（🔍知、📊味、📝美、🔴厳、💜心、✨極）が記載されている

**改善点:**

1. **バージョンの明示がやや弱い** — 方程式 v3.0とルーブリック v2.0にはバージョンがあるが、マニフェストと北極星にはない。全ファイルにバージョンか最終更新日を記載すると追跡性が上がる。
2. **「📖 読む順序」と「📂 ファイル一覧」のマッピングが暗黙的** — 読む順序のStep 1〜4が、ファイル一覧のどの行に対応するかは読者が自分で繋げる必要がある。リンクを直接埋め込むか、Step番号をファイル一覧にも付与すると完全に一致する。
3. **レビュー履歴が「読む順序」に含まれていない** — これは正しい判断（レビューは参考資料）だが、「これらは開発過程の記録であり、初読者は読む必要はない」の一文があると親切。

### 評価: ⚡ A−

前回「README.md / 00_INDEX.md を作れ」と提案し、その通りに作られた。構造は明確で、初見の読者が迷わない設計になっている。上記3点を改善すれば A+ に到達する。

### 品質評価

**良い点:**
- **定義が簡潔かつ正確** — 各用語1〜2行で本質を捉えている
- **階層関係が示されている** — 例: 「A = Core × Place × Companion」の構造がA値の定義内に記載
- **理論最大値（550pt）が明記** — v2.0で誤記だった500ptが修正されている
- **店舗コードの正式名が対応** — NP=Nuit Polaire等（前回の提案が反映）
- **表形式で一覧性が高い**

**不足している用語（追加推奨）:**

| 用語 | 理由 |
|------|------|
| **BV** | ルーブリック v2.0で新設されたビバレッジ軸。方程式には含まれないがルーブリックの主要カテゴリ |
| **SVD** | 文書全体で使われるが「SAPPORO VIEWTIFUL DINING」の定義が用語集にない |
| **VIEWTIFUL** | VIEW + BEAUTIFUL。方程式v3.0内で哲学的注記として説明されているが用語集にもあるべき |
| **テロワール** | 食材と土地の関係を表す概念。複数箇所で使用 |
| **キャリブレーション** | ルーブリックの使い方ガイドで言及 |
| **RESTAURANT OS** | 文書体系全体を指す名称 |

### 20語は十分か？

**結論: 現段階では十分。ただし上記6語の追加で26語にすると完全性が上がる。**

用語集の目的は「初見の読者が迷子にならないこと」。現在の20語は方程式を理解するための最低限をカバーしている。ただし、ルーブリックv2.0の新要素（BV）と、文書群全体の文脈語（SVD, VIEWTIFUL, RESTAURANT OS）が欠けているのは惜しい。

### 配置の妥当性

用語集が00_INDEX.md内に配置されているのは**正しい判断**。INDEX = 入口であり、入口に辞書があるのは理にかなっている。ただし、用語数が30語を超えたら独立ファイル（`99_GLOSSARY.md`）への分離を検討すべき。

### 評価: ⚡ B+

前回「用語集が存在しない（D評価）」だったものが、20語の実用的な用語集として生まれた。B+→A-への道は上記6語の追加。

## 5. 初見の理解度は改善されたか

### 前回指摘した4つの障壁の改善状況

| 障壁 | v2.0の状態 | v3.0の状態 | 改善度 |
|------|-----------|-----------|--------|
| **用語が未定義** | BOCUSE、LTC、NP/JW等が未定義 | 20語の用語集で主要用語をカバー | ⬆⬆ 大幅改善 |
| **前提知識を要求** | v1.0を知らないと意味不明 | v1.0→v2.0→v3.0の進化表で差分が明示 | ⬆ 改善 |
| **抽象度が高い** | 「ブルーエコノミー」等が唐突 | S-Scoreの説明にSATの原体験（釧路での昆布採り）を接続 | ⬆ 改善 |
| **読む順序が不明** | 4ファイルのどれから読むか不明 | 00_INDEX.mdで明確に指示 | ⬆⬆ 解消 |

### 初見シミュレーション（新入社員が初めて読む場合）

**v2.0の場合:**
1. フォルダを開く → 4ファイルが並んでいる → どれから？ → とりあえず方程式を開く → BOCUSEって誰？ → LTCって何？ → 離脱

**v3.0の場合:**
1. フォルダを開く → 00_INDEX.mdを見つける → 「📖 読む順序」を確認 → Step 1のマニフェストへ → 哲学を理解 → Step 2の方程式へ → わからない用語はINDEXの用語集を参照 → 30秒サマリーで全体像を把握 → 詳細を読み進める

**v3.0は「迷子にならない」設計になっている。** これは決定的な改善。

### 残存する理解の壁

1. **掛け算構造の直感的理解** — 「なぜ足し算ではなく掛け算か」の説明がまだ薄い。「FDが0ならCoreも0」という含意を1行で示すとさらに明快
2. **Place係数の決定方法** — 「誰が」「いつ」「どのように」係数を決めるのかが不明。ルーブリックにPlace係数の採点基準がない
3. **BVの位置づけ** — ルーブリックv2.0でBV（ビバレッジ）が新設されたが、方程式v3.0のどこに位置するかが不明確。FD①に内包されるのか、独立カテゴリか？

### 評価: ⚡ B+（前回C+ → 2段階アップ）

初見の読者が「入口」を見つけ、「道順」に従い、「辞書」を参照しながら読み進められる構造が完成した。上記3つの残存壁を解消すればA評価に到達する。
