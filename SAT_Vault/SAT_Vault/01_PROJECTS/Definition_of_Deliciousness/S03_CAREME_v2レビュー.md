## 1. 4ファイルの関係性 — 読む順序は明確か？

### 現状の問題

**読む順序がどこにも書かれていない。** 4ファイルはフォルダ内にフラットに並んでおり、初めて開いた人は「どれから読めばいいのか」がわからない。

### ファイル間の暗黙の依存関係

```
SATのマニフェスト（WHY — なぜこの方程式が必要か）
    ↓
美味しさの定義 v2.0（WHAT — 方程式そのもの）
    ↓
ルーブリック v1.0（HOW — 方程式の使い方）
    ↓
AsiaBest ベンチマーク（WHERE — 世界の中でのSVDの位置）
```

この「WHY → WHAT → HOW → WHERE」の流れは論理的に美しいが、**どのファイルにもこの構造図がない。**

### 提案

- フォルダ直下に `README.md`（または `00_INDEX.md`）を置き、読む順序と各ファイルの役割を1ページで示す
- 各ファイルの冒頭に「前提として読むべきファイル」「次に読むべきファイル」のリンクを設置

## 4. 統合 vs 分割 — どちらが正しいか？

### 結論: **分割のまま + INDEX ファイルの新設 が最善。**

### 理由

| 観点 | 統合（1ファイル） | 分割（現状+改善） |
|------|-----------------|-----------------|
| 一覧性 | ○ 全体を見渡せる | △ ファイル間の移動が必要 |
| メンテナンス性 | ✗ 1人が編集すると全体にロック | ○ 並行編集が可能 |
| 読者の負荷 | ✗ 巨大文書は読まれない | ○ 必要な章だけ読める |
| 外部提供 | ✗ 選択的な提供ができない | ○ 必要なファイルだけ共有可能 |
| Obsidian活用 | ✗ リンクの利点を殺す | ○ Wiki的なナビゲーションが活きる |

## 5. マニフェストの位置づけ — 「前」か「付録」か？

### 結論: **方程式の「前」に置くべき。序文として。**

### 理由

1. **マニフェストは「WHY」を語る文書**。方程式（WHAT）を理解するには、まず「なぜこの方程式が必要か」を知る必要がある。
2. **感情 → 論理 の順序が人間の認知に合う**。マニフェストで心を動かし、方程式で頭を納得させる。逆順（論理→感情）では、方程式が「ただの数式」に見えてしまう。
3. **「僕のじゃなくて、この街のなんだよ」を最初に読むか最後に読むかで、方程式の⑤Place係数の意味が根本的に変わる。** 最初に読めば「SVDの魂」として染み込む。最後に読むと「補足説明」に格下げされる。
4. **アントナン・カレームの教え** — カレームは「料理は建築である」と言った。建築に土台が必要なように、方程式にはマニフェストという土台が必要。

### ただし1点の注意

マニフェストは「生の声」であるがゆえに、**技術文書として読むと浮く**。序文として配置する場合は、以下の処理が有効：

- タイトルを「SATのマニフェスト」→「**序文 — この方程式に込めた想い**」に変更
- BOCUSEへの回答セクションは本文（v2.0）側に移動
- タイムスタンプは残す（創業者の生の声であることの証として）

## 6. RESTAURANT OS として他社に提供する場合の文書デザイン提案

### 現状の課題

現在の文書群はSVD内部のナレッジベース（Obsidian Vault）として書かれている。他社に提供するには以下の変換が必要：

### デザイン提案: 3層構造

```
┌─────────────────────────────────────────────┐
│  Layer 1: RESTAURANT OS — Public Edition    │
│  ─────────────────────────────────────────  │
│  汎用フレームワークとしての方程式・ルーブリック │
│  SVD固有の情報を抽象化                        │
│  → PDF / Web で配布                          │
├─────────────────────────────────────────────┤
│  Layer 2: SVD Implementation Guide          │
│  ─────────────────────────────────────────  │
│  SVD固有の係数テーブル、店舗コード、戦略       │
│  マニフェスト、ベンチマーク分析               │
│  → 社内 Obsidian Vault                      │
├─────────────────────────────────────────────┤
│  Layer 3: Operational Playbook              │
│  ─────────────────────────────────────────  │
│  各店舗の日常運用マニュアル                    │
│  PDCA記録、個人評価シート                     │
│  → 各店舗の現場文書                          │
└─────────────────────────────────────────────┘
```

### Layer 1 の文書デザイン要件

| 要素 | 提案 |
|------|------|
| **タイトル** | 「RESTAURANT OS — 美味しさを科学する」 |
| **ページ構成** | 序文(1p) → 概要(2p) → 方程式(4p) → ルーブリック(8p) → 導入ガイド(2p) |
| **デザイン言語** | ミニマル和モダン — 余白を活かした日本的美意識 |
| **フォント** | 見出し: Noto Sans JP Bold / 本文: Noto Serif JP |
| **カラーパレット** | 墨色(#2C2C2C) × 白(#FAFAFA) × 差し色に北海道の海(#1B4965) |
| **図版** | 方程式の構造図をインフォグラフィック化 |
| **判型** | A4横（プレゼンテーション兼用） |

### 汎用化のために必要な変換

| SVD固有の要素 | 汎用化 |
|--------------|--------|
| 「北海道の海と大地」 | 「あなたの土地のテロワール」 |
| Place係数テーブル（藻岩山等） | 「各社が定義する場所性テーブル」テンプレート |
| Companion係数（LTC連動） | 「シーン別体験係数」テンプレート |
| BOCUSE条件への参照 | 削除（内部開発プロセスのため） |
| 店舗コード（NP/JW/BQ/GA） | 「導入事例」として匿名化 or SVDケーススタディ |

### ネーミング提案

外部提供時のプロダクト名候補：

1. **RESTAURANT OS** — シンプル、テック感あり
2. **Deliciousness Framework** — 直球、国際展開向き
3. **A-Score System** — 方程式のAを活かしたブランド名
4. **VIEWTIFUL Standards** — SVDブランドを前面に出す場合
