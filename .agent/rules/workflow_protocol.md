# SVD Workflow Protocol

> 統合憲法の精神のもと、日々の開発・業務における行動指針。

---

## 1. 計画優先の原則

**計画モード発動条件**（以下のいずれか）:
- 3ステップ以上/アーキテクチャ影響/2+ファイル変更/API影響

**即時再計画**: 想定通りに進まない → 無理に押し進めず、問題を言語化 → 原因特定 → 新計画

**検証も計画に含める**: 「何をもって完了とするか」を事前に明文化

## 2. コンテキスト管理

- サブエージェント活用: 調査はサブエージェントに委譲（1タスク1エージェント）
- **KI確認**: 新しい調査の前に必ずKIを確認、既存があればベースに拡張

## 3. 学習ループ

**lessons.md** (`/Users/satoshiiga/dotfiles/.agent/lessons.md`): SATから修正を受けたら「何が/なぜ/次回どう防ぐ」を記録。同じミス2回で `rules/` に昇格。

## 4. 検証の原則

完了前に必ず動作を証明する（テスト/ログ/ブラウザ確認）。

自問チェック:
- SATがチームにデモして自信を持って見せられるか？
- シニアエンジニアがコードレビューで承認するか？

## 5. エレガンスと実用性

| 変更規模 | アプローチ |
|---|---|
| 単純（typo, 1行修正） | そのまま直す |
| 中程度（機能追加, UI調整） | 一度立ち止まり、より良い方法を考える |
| 大（3+ファイル, API変更） | 必ず計画を立て、エレガントな設計を追求 |

ハック禁止: 一時的な修正は技術的負債。根本原因を解決する。

## 6. 自律バグ修正

SATからバグ報告 → 自分でログ確認 → 原因特定 → 修正 → 動作確認 → 結果報告。SATにハンドホールディングを求めない。

## 7. ロールバック安全性

- 破壊的変更前に `git commit` → 不確実ならブランチ
- データ変更時はオリジナル保持、前後の件数/合計値を照合

## 8. タスクフロー

```
計画 → SAT確認 → 実行 → 進捗更新 → 検証 → 完了報告
```

| 成果物 | 配置先 |
|---|---|
| タスク | `brain/<id>/task.md` |
| 実装計画 | `brain/<id>/implementation_plan.md` |
| 完了報告 | `brain/<id>/walkthrough.md` |
| 学習記録 | `.agent/lessons.md` |
| 永続ルール | `.agent/rules/` |

---

**核心**: 簡潔さ第一 / 怠惰の禁止 / 最小影響 / SATの時間を守る
